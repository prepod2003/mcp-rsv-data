# MCP:RSV Data — мост (Go)

> Локальный мост-посредник между MCP-клиентом (Claude, Cursor и др.) и базой 1С.
> Подключение **через COM (внешнее соединение)** — веб-публикация HTTP-сервиса **не нужна**.
> Отдельный компонент продукта MCP:RSV Data. Лицензия — MIT.

## Зачем мост

MCP-клиент умеет запускать локальную программу и общаться с ней по **stdio**, но не умеет сам
ходить в базу 1С. Мост принимает stdio от клиента и зовёт диспетчер расширения в базе через COM:

```
MCP-клиент  ──stdio (JSON-RPC)──►  rsvdata-bridge  ──COM──►  1С (внешнее соединение)
                                                              └─ RSVData_Сервер.ОбработатьСообщение(текст)
```

Что это даёт:

- **Без веб-публикации.** Для программиста 1С это убирает самую болезненную часть настройки.
- **Удобный вход.** Логин/пароль задаются в мастере и хранятся локально — никакого `base64` в конфиге клиента.
- **stdio-транспорт** для клиентов, которым он нужен.

## Сторона 1С — менять ничего не нужно

Диспетчер `CommonModule.RSVData_Сервер.ОбработатьСообщение(ТекстСообщения)` уже экспортный и имеет
флаг **«Внешнее соединение»** (как и `RSVData_Ядро`, `RSVData_Справка`). Поэтому мост зовёт его
напрямую — отдельный модуль-адаптер не нужен. Достаточно установленного расширения RSVData в базе.

## Сборка

Нужен Go 1.21+.

```bash
cd bridge
go build -o rsvdata-bridge.exe .
```

Зависимость одна — [`github.com/go-ole/go-ole`](https://github.com/go-ole/go-ole) (COM на Windows).
Платформа: **Windows** (COM). Кроссплатформенность (Linux/macOS) — на будущее.

## Подключение для пользователя (мастер)

```bash
rsvdata-bridge.exe setup
```

Мастер сначала спрашивает **способ подключения** и покрывает оба:

- **[1] HTTP-сервис 1С** — спрашивает **клиента** (Claude Code/VS Code, Cursor, Cline), адрес сервиса,
  логин и пароль, проверяет связь HTTP-запросом и **сам собирает заголовок `Authorization: Basic <base64>`**
  (Base64 руками не считают). Печатает готовый блок `.mcp.json` с правильным для клиента типом транспорта
  (`http` / `streamable-http` / `streamableHttp` — у клиентов он разный!). Конфиг моста при этом не
  сохраняется (в этом режиме мост в рантайме не нужен — клиент ходит по HTTP сам).
- **[2] Локальный мост (COM)** — спрашивает имя подключения, тип базы (файловая/серверная), путь или
  сервер+имя, логин и пароль, проверяет связь через COM (ping) и сохраняет настройки в **отдельный файл
  по имени** `%APPDATA%\MCP-RSV-Data\<имя>.json` (права 0600). Печатает блок `.mcp.json` с `type: stdio`
  и `args: ["serve", "--config", "<путь к этому файлу>"]`.

**Несколько баз через мост.** Один бинарник обслуживает любое число баз: запускайте мастер по разу на
каждую с разным именем — получится по файлу `<имя>.json` и по записи в `.mcp.json` (каждая со своим
`--config`). Нужную базу мост выбирает по файлу из `--config`; без `--config` берётся `bridge.json`.

В обоих случаях логин/пароль в конфиг клиента вписывать не нужно (для HTTP они уже в готовом заголовке,
для COM — в файле настроек моста).

**Авто-запись в конфиг клиента.** В конце мастер предлагает дописать сервер в конфиг клиента сам:
спрашивает путь к файлу (`.mcp.json`, `cline_mcp_settings.json` и т. п.), читает его, добавляет/заменяет
запись в `mcpServers` (сохраняя остальные серверы и прочие ключи), пишет обратно — JSON собирает
программа, ручная вставка фрагмента с запятыми не нужна. Прежний файл бэкапится в `<файл>.bak`; невалидный
JSON не трогается. При отказе — обычный путь: блок напечатан, сохранён в файл и в буфер обмена.

## Команды

| Команда | Назначение |
|---|---|
| `rsvdata-bridge` (без аргументов) | Мастер настройки — **по умолчанию**, рассчитан на запуск двойным щелчком. |
| `rsvdata-bridge setup` | То же явно. |
| `rsvdata-bridge serve` | Режим моста. Запускается MCP-клиентом: stdin → COM → stdout. |
| `rsvdata-bridge ping`  | Разовая проверка связи через COM (диагностика). `--method tools/list` и т.п. |
| `rsvdata-bridge diag`  | Список расширений базы + доступность модулей RSVData через внешнее соединение. |

Общие флаги: `--config <путь>` (файл конфига), `--connect "<строка соединения 1С>"` (переопределяет конфиг).

## Как устроено (для разработчика)

- `main.go` — разбор подкоманды.
- `config.go` — конфиг (структурные поля → строка соединения `File=…`/`Srvr=…;Ref=…;Usr=…;Pwd=…`), пути, чтение/запись.
- `onec.go` — COM: выделенный поток ОС (`runtime.LockOSThread` + `CoInitializeEx`), `V83.COMConnector`,
  одно открытое соединение на весь сеанс, вызов `RSVData_Сервер.ОбработатьСообщение`.
- `serve.go` — stdio-цикл: построчное чтение JSON-RPC, **пропуск уведомлений** (пустой ответ), вывод
  **компактного** ответа одной строкой (`json.Compact` — 1С отдаёт многострочный JSON, что сломало бы
  построчное кадрирование stdio). Диагностика — в stderr.
- `setup.go` — мастер + подкоманда `ping`.

Тонкости COM проверены смоком (см. ниже).

## Проверка (смок)

End-to-end через COM на файловой базе подтверждён: `ping` → `result`, `tools/list` → инструменты,
`tools/call describe`/`query` → реальные данные. Уведомление (`notifications/initialized`) корректно
без ответа; все ответы — компактные однострочные.

Локальная проверка:

```bash
rsvdata-bridge.exe ping --connect 'File="D:\путь\к\базе";Usr="Администратор";Pwd="";'
```

> Если `setup`/`ping` падают на «не удалось создать V83.COMConnector» — на машине не зарегистрирован
> COM-соединитель платформы: выполнить `regsvr32 "<путь к 1С>\bin\comcntr.dll"` (разрядность 1С должна
> совпадать с разрядностью моста — собираем под Windows amd64).
